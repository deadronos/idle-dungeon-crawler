# 001 - Player Classes and Party System

**Date:** 2026-03-10
**Status:** Superseded by [005 - Expandable Party System](005-expandable-party-system.md)

## Context

> This record captures the original introduction of the party-based autobattler structure. The current shipped starting-party and recruitment rules are defined in [005 - Expandable Party System](005-expandable-party-system.md).

The game started as a simple clicker game where the player clicked an enemy to deal damage. To create a deeper and more engaging RPG experience, we needed a system that allowed for varied gameplay styles, strategic team composition, and idle progression. We needed to transition from an active clicker to an autobattler.

## Decision

We initially implemented a three-hero starter Party System where the player chose a lead hero and entered the dungeon with a full adventuring team.

1. **Classes:** We introduced distinct hero classes (Warrior, Cleric, Archer), each with unique attributes, derived stats, and secondary resources. (See [002 - Attributes and Derived Stats](002-attributes-and-stats.md) for formulas).
   * **Warrior:** Uses Rage (generated by attacking), focuses on Strength and Vitality, and spends that Rage on heavy weapon skills.
   * **Cleric:** Uses Mana (regenerates slowly), focuses on Intelligence and Wisdom, and prioritizes healing injured allies when the party is under pressure.
   * **Archer:** Uses Cunning (rapid regeneration), focuses on Dexterity and critical damage, and spends Cunning on empowered precision shots.
2. **Attributes:** We defined a core set of attributes: Constitution (VIT), Strength (STR), Dexterity (DEX), Intelligence (INT), and Wisdom (WIS). (See [002 - Attributes and Derived Stats](002-attributes-and-stats.md)).
3. **Active Time Battle (ATB):** We replaced manual clicking with an ATB system. Every entity (player and enemy) has an action progress bar that fills over time, triggering attacks or skills when full. (See [003 - Combat Loop and ATB Mechanics](003-combat-and-atb.md)).
4. **Floor Navigation, Autofight, and Autoadvance:** We support explicit floor navigation controls so the player can step backward or forward one floor at a time. `Autofight` determines whether the ATB combat simulation is actively resolving the current encounter. `Autoadvance` separately determines whether a victory should automatically move the party to the next floor. This allows the player to pause on a floor their current party can reliably farm.
5. **Party Wipe:** If the entire party dies, they are fully healed but reset to Floor 1, losing all current gold (a rogue-lite prestige mechanic foundation). (See [004 - Progression, Leveling, and Scaling](004-progression-and-scaling.md)).
6. **Original Starting Party:** Character creation originally chose the party leader's name and class, then automatically filled the rest of the starter party with the remaining two classes so every run began with a balanced trio. This behavior was later replaced by the expandable-party progression model in [005 - Expandable Party System](005-expandable-party-system.md).

## Consequences

* **Easier:** Designing new classes or enemies is straightforward by adjusting base attributes and resource types. The game can now be played entirely idly.
* **Difficult:** Balancing becomes much more complex, requiring careful tuning of attribute multipliers, exponential EXP curves, and enemy scaling to ensure the game doesn't become too easy or impossible at higher floors. It also requires more complex UI to display party state.
